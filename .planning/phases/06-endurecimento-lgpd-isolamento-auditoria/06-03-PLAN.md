<objective>
DATA-02 vertical slice — the data-subject rights (export everything + delete everything), including the single most dangerous operation in the app. After this plan, a real user can, from the Privacidade e conta screen: download a complete JSON bundle of all their data, and permanently delete their account (DB + Storage + auth) behind a type-to-confirm "APAGAR" gate that signs them out.

Two server primitives ship here: `exportMyData` reads all 14 owned tables via the RLS-scoped server client (so "only my rows" is structural, not filter-dependent) and returns a JSON bundle with embedded transactions + MEI CSVs; `deleteMyAccount` uses the server-only service-role `admin.ts` (06-01) to remove Storage `{user_id}/` objects FIRST, then `auth.admin.deleteUser` LAST (the `ON DELETE CASCADE` schema deletes all 14 tables automatically). This plan flips the three deferred Wave-0 tests (lgpd-export, lgpd-delete, lgpd-delete-isolation) GREEN.

Purpose: deliver DATA-02 end-to-end; prove the delete is complete (A fully gone) AND surgical (B untouched).
Output: `bundle.ts`, `export-data.ts`, `delete-account.ts` (+ test), `ExportDataButton`, `DeleteAccountForm`, the completed Conta screen, and 3 GREEN LGPD tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → delete action | the UI imports ONLY the Server Action; the service-role admin client stays server-only |
| session claims → userId | export + delete derive userId from getClaims, never from request input |
| Storage (not FK-cascaded) → delete | Storage must be removed explicitly before the auth delete |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure / Elevation | admin.ts (service-role) imported transitively by a client | mitigate | delete-account.ts imports admin.ts server-side; the UI imports ONLY the action; `server-only` fails the build on a client import; bundle-secret audit (06-04) stays green |
| T-06-09 | Spoofing / Tampering | forged userId in delete/export | mitigate | userId from getClaims (session) only; never read from input; confirm `z.literal('APAGAR')` |
| T-06-10 | Repudiation / Integrity | orphaned data after a partial delete | mitigate | Storage-remove FIRST (idempotent), auth.admin.deleteUser LAST; a Storage failure leaves the account intact + retryable; lgpd-delete-isolation proves B untouched |
| T-06-11 | Tampering | hand-ordered per-table DELETE racing RESTRICT FKs | mitigate | use the auth.users `ON DELETE CASCADE` (atomic, ordering-correct); NO manual delete loop (Anti-Pattern) |
| T-06-03 | Information Disclosure | export missing a table (incomplete LGPD) | mitigate | bundle iterates OWNED_TABLES; lgpd-export asserts every owned table is a bundle key |
| T-06-SC | Tampering | npm installs (e.g. a zip lib) | mitigate | single-JSON bundle — ZERO new packages (06-RESEARCH A1); if a zip lib is ever wanted, run the legitimacy gate first |
</threat_model>
