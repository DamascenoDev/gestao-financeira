<root_cause>
G-04: `src/lib/adherence.ts` maps teto `< 80%` → status `no-limite` with label "No limite". The label is misleading for a low-utilization teto. The 80%/100% thresholds are correct; only the calm-band COPY (label) is wrong.

G-03 — diagnose before fixing. The migration `supabase/migrations/0014_adherence_views.sql` ALREADY drives the period set off INCOME (the `base` CTE joins each meta to every income month; consumo realized is a LEFT join onto `v_category_totals`, so a zero-spend consumo meta yields `realized_cents = 0` and a row). Therefore a zero-spend teto SHOULD materialize **whenever the user has income that period**. The dashboard's `.eq('month_key', mes)` then keeps it. Two residual client-side possibilities the executor must check locally:
  (a) `buildRows` (dashboard/page.tsx) iterates `viewRows` and includes consumo rows unconditionally — confirm a realized-0 consumo row is NOT filtered out anywhere in the page mapping.
  (b) The `RowList.semReceita` gate (`incomeCents <= 0n || rows.every(r => r.adherenceBp === null)`) hides the LIST when there is no income — which is correct (the % meta is undefined without income), but verify it does not also blank a period that DOES have income + a zero-spend meta.
The production symptom is the spend-driven omission; reproduce it LOCALLY (a teto meta + income + zero spend in that category) and confirm the row appears. If it does NOT appear locally, the bug is in the page mapping (fix it there); if it appears locally but not in prod, the production view is simply stale (the redeploy/db push in the downstream live cycle resolves it) — record that finding in the SUMMARY. Do NOT add a migration unless local repro proves the view itself drops the row.

G-02: `src/components/adherence-row.tsx` column 1 is `<div className="flex min-w-0 items-center gap-2">` holding `<CategoryBadge>`. `CategoryBadge` (category-badge.tsx) wraps the name in `<span className="truncate">` but the badge root is `inline-flex` WITHOUT `min-w-0`, so `truncate` has no effect and a long name overflows into the bar column.
</root_cause>
