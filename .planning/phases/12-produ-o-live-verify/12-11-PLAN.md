<root_cause>
`grep 'type="date"' src/` finds native date inputs in SIX components: `nf-form.tsx`, `abastecimento-form.tsx`, `saida-form.tsx`, `mei-settings-form.tsx`, `receita-form.tsx`, `transacao-form.tsx`. The native `<input type=date>` display order follows the browser/OS locale (en-US → MM/DD/YYYY) and is NOT reliably forceable per-app via CSS. Each of these holds an ISO `yyyy-MM-dd` string in state and the server actions validate the `/^\d{4}-\d{2}-\d{2}$/` shape (see receita-form.tsx onSubmit, saida-form.tsx validate, incomes.ts createAdhocIncome). Table/label DISPLAYS already render pt-BR (e.g. nf-table.tsx `ddMM`), so only the INPUT fields regressed.

Approach (masked text input — lowest risk, no calendar dependency): a controlled `<input inputMode="numeric">` that shows/edits `dd/mm/aaaa`, masks digits as the user types, and converts to/from an ISO `yyyy-MM-dd` value at the prop boundary. `value` prop = ISO (or ''), `onChange` emits ISO (or '' while incomplete/invalid). This keeps the existing ISO state + server validation untouched. Mirror `money-input.tsx`'s controlled forwardRef pattern. (`react-day-picker` calendar + `popover` are available as an alternative, but the masked field is the smaller, dependency-free change and matches the existing manual-typing UX.)
</root_cause>
