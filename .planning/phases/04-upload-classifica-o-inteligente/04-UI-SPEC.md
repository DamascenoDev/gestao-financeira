# Phase 4 — UI Design Contract

> Visual and interaction contract for the highest-risk slice: **ingestão de faturas (OFX/CSV) → parse → dedup → classificação por memória → revisão que aprende**. Scope: a Upload screen, a CSV column-mapping dialog, and the central **import review** screen (dense pre-persist grid reusing the Extrato patterns + SelectionActionBar + ReservaPicker), plus dedup feedback.
>
> **This phase INHERITS the app-wide design system established in Phase 2** (`02-UI-SPEC.md`) and extended in Phase 3 (`03-UI-SPEC.md`): teal `--primary` accent (reserved), OKLCH semantic money tokens (`--income`/`--expense`/`--allocation`/`--consumption`/`--destructive`), `--font-mono` tabular numbers, dense table layout, `base-nova` shadcn, pt-BR copy, money only through `src/lib/money.ts`. This spec does **NOT** redefine that system — it extends it for the new screens. Where this doc is silent, the Phase 2/3 contracts govern.
>
> **AI is DEFERRED for v1** (CONTEXT `scope_decision`). Phase 4 ships the full memory-first + manual-on-miss pipeline; the LLM suggestion step is a clean, pluggable seam that returns nothing today. The review UX is designed so a **memory-miss (unclassified) row is visually obvious and one-tap classifiable**, with a labeled, empty affordance where an AI suggestion will later slot — **build the slot, not the call.**
>
> Generated by gsd-ui-researcher. Stack locked: Next.js App Router, TS strict, Tailwind v4 (`@theme`), shadcn `base-nova`, lucide, pt-BR. Parse runs server-side; uploads go browser→Storage by signed URL.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `components.json` present, `registries: {}`) |
| Preset | `base-nova` (inherited) |
| Component library | Radix via base-nova primitives (vendored) |

| Font | `--font-sans` (Geist) for prose; `--font-mono` (Geist Mono) for all money, counts, dates |

**Design intent carried forward.** The review screen is the most data-dense surface in the app — it must read as the Extrato's twin: calm, monochrome chrome, color only where it carries meaning. The *one* new piece of signal this phase introduces is **"this row needs your attention"** (a memory miss). That signal must be unmissable without becoming alarming — it is a to-do, not an error. We use the calm `--consumption` amber + a text label + an icon, **never** `--destructive` red (red stays reserved for genuine failures: wrong file type, parse failure, over-limit). The future AI suggestion lives in a reserved, labeled, currently-empty slot inside the unclassified state — visible as intent, inert in v1.

**Accent (teal) reserved for:** primary action button (one per surface — "Selecionar arquivo" on upload, "Confirmar importação" on review, "Salvar mapeamento" in the CSV dialog), the dropzone **drag-over** state, active nav item, focus rings, checked selection controls. **Never** for the unclassified attention signal (amber owns that), never for origem badges (those are neutral/muted), never as a generic link.

**Classification-origin color rule:** origem is **information, rendered in neutral/muted by default** — `memória` and `manual` are calm, low-contrast `badge`s (`bg-muted text-muted-foreground`) because a *classified* row is the quiet, resolved state. Only the **unclassified** state escalates to amber `--consumption`. The future **sugerida** origin reserves a `--primary`-tinted badge with a `sparkles` glyph (defined here, not rendered in v1).

60/30/10 holds: the review screen is overwhelmingly neutral table chrome; amber appears only on the subset of rows that need attention; teal appears only on the single confirm CTA + selection state; green/indigo only on the CategoryBadge of already-classified rows.

### 1. Upload de fatura (`(app)/importar`)

Upload OFX/CSV direct to private Storage by signed URL; server parses; land in review. The function only ever receives the Storage **path**, never the file bytes (signed/resumable upload bypasses the 4.5MB function limit).

- **Header:** h1 "Importar fatura" + a one-line helper "Suba o OFX ou CSV do seu banco. Vamos extrair, deduplicar e pré-classificar pela memória." Optional: a muted link "Como exportar do meu banco?" (out of scope to build the help content — just the affordance, or omit).
- **Body:** the `UploadDropzone` (the only primary surface). Primary CTA inside it: **"Selecionar arquivo"** (teal) + "ou arraste aqui". Format hint (`text-xs muted`): "OFX (recomendado) ou CSV · até {limite}".
- **On file chosen → uploading:** swap the dropzone for `UploadProgress` (filename + `progress` bar + "Enviando…"). After upload completes → "Processando…" (indeterminate, server parse) → on success **redirect to review** (`/importar/[statementId]`).
- **CSV ambiguity branch:** if the uploaded file is CSV and headers are ambiguous, after parse the **`CsvColumnMapper` dialog** opens (screen 2) before review. OFX skips this entirely.
- **States:**
  - **Idle / empty:** the dropzone is the empty state — no separate `empty` primitive needed; the dropzone copy *is* the call to action.
  - **Uploading / parsing:** `UploadProgress` (Enviando… / Processando…). Keep the user on the page; do not navigate until parse resolves.
  - **Error — wrong type:** inline `text-destructive` under the dropzone: "Formato não suportado. Envie um arquivo OFX ou CSV." (reject client-side before upload).
  - **Error — too large:** "Arquivo muito grande. O limite é {limite}." (client-side).
  - **Error — parse failure:** after upload, if parse yields nothing/throws: a `text-destructive` block "Não foi possível ler este arquivo. Verifique se é um extrato OFX/CSV válido e tente de novo." + a "Tentar outro arquivo" button (resets to idle). (Steer to OFX when CSV repeatedly fails — STACK guidance.)
  - **Loading (page):** `skeleton` for the dropzone block is unnecessary (renders instantly); the *parse* wait is covered by `UploadProgress`.

### 2. CSV column-mapping dialog (`CsvColumnMapper`)

Fires only when a CSV's headers can't be confidently auto-mapped. A small, guided mapper that produces a reusable **perfil**.

- **Trigger:** post-parse ambiguity on a CSV upload (screen 1). OFX never triggers it.
- **Layout (`dialog`, `lg` padding, single column):**
  - Title "Mapear colunas do CSV" + description "Indique qual coluna é a data, a descrição e o valor. Salvamos esse mapeamento para os próximos arquivos do mesmo layout."
  - Three `field` rows, each a `Select` of the detected column headers → **Data**, **Descritor**, **Valor**. Each Select shows the column's first sample value as helper (`text-xs muted`) so the user maps by example.
  - A **preview table** (shadcn `table`, max ~5 rows) rendered *as parsed*: Data `dd/MM` mono · Descritor (truncate) · Valor `formatCents` mono right. Updates live as the mappings change.
  - A "Salvar como perfil" `switch`/checkbox (default on) + optional profile name (`input`, defaults to the bank/header signature).
- **Validation:** Data, Descritor, Valor must each map to a **distinct** column; block "Salvar mapeamento" until all three are set and distinct. Inline `FieldError` if a parsed sample value fails (e.g. valor não numérico → "Esta coluna não parece conter valores monetários.").
- **Primary CTA:** **"Salvar mapeamento"** (teal) → re-parses the CSV with the mapping (and persists the profile) → opens review.
- **Secondary:** "Cancelar" → discards (back to upload idle; the uploaded file stays in Storage but no transactions are created).
- **States:** **empty preview** (no rows parse with the current mapping) → inline muted "Nenhuma linha foi reconhecida com este mapeamento. Verifique as colunas." **Profile reuse:** if a saved profile matches the header signature, skip the dialog entirely and go straight to review (surface a quiet toast "Mapeamento reaproveitado").

### 3. Tela de revisão da importação (`(app)/importar/[statementId]`) — the core screen

A dense pre-persist grid of parsed transactions. **Nothing is written to `transactions` until the user confirms.** Reuses the Extrato grammar + SelectionActionBar + ReservaPicker.

- **Header:** h1 "Revisar importação" + `ImportSummaryHeader` (counts) + statement context (banco/arquivo + período `dd/MM`–`dd/MM`, muted). Primary CTA **"Confirmar importação"** (teal, top-right and/or sticky bottom).
- **Summary header (`ImportSummaryHeader`):** "**N** transações · M novas · **K** não classificadas · J duplicadas ignoradas". The "K não classificadas" is amber + `triangle-alert` when `K>0` and is **click-to-filter** ("ver não classificadas" → filters the grid to memory-miss rows). See §Dedup for the re-upload "0 novas" copy.
- **Grid (`ImportReviewTable`):** columns per §Component Inventory. Sorted by Data desc default. **Memory-miss rows visually distinct** (amber left-border + "Não classificada" Origem badge) and float to attention via the filter affordance — they are the user's to-do list.
- **Classify a single row:** click the Categoria cell → inline `CategoryBadge` Select (reused from `InlineCategoryCell`). Choosing the **Reserva** category opens the focused **"Qual reserva?"** dialog (`ReservaPicker`) — identical to the Extrato re-tag flow; on confirm the row is flagged as an aporte (the actual ledger entry is created at **Confirmar**, not before). Origem flips `não classificada → manual`, amber accent drops.
- **Bulk classify (the load-bearing reuse):** select ≥1 memory-miss row (header select-all, per-row, shift-range) → the **`SelectionActionBar`** appears (sticky `--card`): "{n} selecionadas" · category Select · **"Classificar"** · "Limpar seleção". (The bar's button copy is "Reclassificar" in the Extrato; here it reads **"Classificar"** since these rows start unclassified — pass it as a prop or accept the inherited label; prefer "Classificar" for accuracy.) The Reserva category is **excluded** from the bulk picker (same rule as Extrato — bulk can't collect a per-row reserva). Applying updates origem to `manual` for the selection.
- **Suggestion slot (deferred AI):** each unclassified row's Categoria cell reserves the `SuggestionSlot` — inert in v1 (renders "—"/nothing). When AI ships, it renders an "Aplicar sugestão: {Categoria}" chip. **Design the cell so adding the chip later does not reflow the column.**
- **Recorrente flag:** rows whose normalized descritor recurs across months show the `RecorrenteTag` inline in the Descritor cell. It is informational at review time (no action required); it persists onto the transaction/merchant.
- **Confirm / Persist (`Confirmar importação`):** writes the parsed rows to `transactions`, **learns the patterns** (merchant→categoria for every classified row; merchant→reserva + creates the reserva aporte ledger entry for Reserva rows — RSV-06), and marks the statement done. Success → `sonner` "{M} transações importadas" + redirect to the Extrato (at the dominant month) or a short summary. Learning happens **only here**, on human confirm (never at parse).
- **Unclassified-on-confirm guard:** if `K>0` when the user hits Confirmar, do **not** hard-block — show an `alert-dialog`: "Há {K} transações não classificadas. Elas serão importadas sem categoria e você pode classificá-las depois no extrato. Confirmar mesmo assim?" [Voltar] [Confirmar]. (Memory only learns from *classified* rows, so unclassified ones simply persist category-less — honest, not lost.)
- **States:**
  - **All classified (memory hit on everything):** `K=0` — no amber rows; summary reads "{N} transações · {M} novas · 0 não classificadas". The screen is a calm confirm-and-go. This is the "extrato de merchants conhecidos → ~0 chamadas externas" happy path.
  - **Needs attention (`K>0`):** amber rows present; the summary's "K não classificadas" is the entry point. Encourage but don't force classification.
  - **Re-upload / all duplicates (`M=0`):** see §Dedup — header shows the "0 novas" success line; grid shows the duplicate-only empty state (the `empty` primitive): heading "Nada novo neste arquivo" · body "Todas as {J} transações deste arquivo já estão no seu extrato. Nada foi duplicado." · CTA "Voltar para o extrato". No Confirmar needed.
  - **Loading:** `skeleton` review rows (keep the summary header skeleton + h1 visible) while the server parse/classify resolves.
  - **Error (load):** inline `text-destructive` "Não foi possível carregar a revisão. Tente recarregar a página."

### 4. Dedup feedback

How idempotent dedup is surfaced (IMP-01 acceptance: re-upload = "0 novas", never duplicates).

- **Two-layer dedup is invisible plumbing; the UI only reports outcomes:**
  - **File-level (content hash):** re-uploading the exact same file → the system recognizes it and the review/summary reports **M=0 novas, J = all** as duplicates. The user is taken to (or shown) the duplicate-only empty state immediately.
  - **Transaction-level (`dedupe_key`):** in overlapping statements, already-present transactions are silently skipped and counted into **J duplicadas ignoradas**; only genuinely new rows count into **M**.
- **Surfacing:** the `ImportSummaryHeader` is the single source of dedup truth — "{N} transações · {M} novas · {J} duplicadas ignoradas". `J` is muted (not an error — duplicates being ignored is the system working). On `M=0`, the header swaps to the explicit success line **"0 novas — este arquivo já foi importado"** (or "todas as transações já estão no extrato") so the "0 novas" acceptance criterion is literally visible.
- **No silent magic:** the user always sees *how many* were new vs ignored. A `sonner` toast on landing for a pure re-upload: "Este arquivo já foi importado — 0 novas transações."

## Money & Number Formatting

Inherited from Phase 2/3:

- **Money:** every value via `formatCents` (pt-BR `R$ 1.234,56`); CSV/OFX amounts normalized to **integer cents at the parse boundary** (BR specifics: vírgula decimal `1.234,56`, `dd/MM/yyyy`). Mono + `tabular-nums`, right-aligned. Parsed gastos render `kind="expense"`, **neutral, never red**; a parsed credit/estorno (if any) renders `+` `--income`.
- **Counts:** N / M / K / J render as integers, mono `tabular-nums`. Never show a bare unlabeled number — always paired with its noun ("3 não classificadas").
- **Dates:** parse BR `dd/MM/yyyy` explicitly at ingest; display `dd/MM` in the dense grid, period range `dd/MM`–`dd/MM` in the header. Month boundaries via `date-fns-tz` pinned to `America/Sao_Paulo` so a month-end transaction lands in the correct civil month.
- **Descritor:** raw shown as-is (truncate+tooltip); normalized shown mono muted — both are display-only; `normalizeDescriptor` is the source of the matched key, never re-derived in the cell.

## Accessibility Basics

Inherited from Phase 2/3, plus:

- **Dropzone:** the drag area is backed by a real labeled `<input type="file">` (keyboard + screen-reader reachable); drag-over is a visual enhancement, not the only path. `aria-label` "Selecionar arquivo OFX ou CSV". Validation errors are associated text, not color-only.
- **Color never sole signal:** the memory-miss attention state pairs the amber accent with the **text** "Não classificada" + a `triangle-alert` glyph; OriginBadge always shows a text label; RecorrenteTag shows "Recorrente" text. Survives grayscale and color-blindness.
- **Upload progress:** `progress` exposes `role="progressbar"` + `aria-valuenow`; the Enviando…/Processando… transition is announced via an `aria-live="polite"` status, including the indeterminate parse phase.
- **`SuggestionSlot`:** when inert (v1) it is `aria-hidden` / empty (no phantom control). When active (future), the chip is a real button with an accessible name "Aplicar sugestão: {Categoria}".
- **Summary counts:** the "K não classificadas" filter affordance is a real button with an accessible name; clicking it announces the filtered result.
- **"Qual reserva?" dialog & ReservaPicker:** required-field semantics inherited from Phase 3 (accessible name, `aria-invalid`, announced conditional reveal).
- **Confirm guard:** the unclassified-on-confirm `alert-dialog` is focus-trapped, escape-cancellable.
- **Contrast:** amber `--consumption` attention text ≥4.5:1 on its surface; the left-border accent + badge are ≥3:1 non-text indicators, in light and dark (tokens tuned in Phase 2).

## Deferred-AI Seam (UI summary)

The LLM suggestion is **not built**; the UI reserves its exact slot so adding it later is additive, not a rework:

- `OriginBadge` defines a **`sugerida`** variant (`--primary`-tinted + `sparkles`) — never rendered in v1 (`suggestCategory()` returns `null`).
- `SuggestionSlot` reserves space in the unclassified Categoria cell — inert/`aria-hidden` in v1; renders the "Aplicar sugestão: {Categoria}" chip when a non-null suggestion arrives. The column must **not reflow** when the chip appears.
- No copy, color token, or layout depends on the AI being present: the screen is fully functional as memory-first + manual-on-miss. Adding AI only fills the reserved slot + flips eligible rows' origem to `sugerida`.
