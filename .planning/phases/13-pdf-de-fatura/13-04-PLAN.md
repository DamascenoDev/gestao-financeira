<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client review grid -> confirmImport | the grid is un-persisted client state; deletes only shrink the confirm payload — nothing lands in `transactions` until confirm (PDF-03) |
| server summary -> client counters | `descartadas`/`duplicadas` are server-derived (persisted `statements.summary`); the client must not recompute them from a length delta (correctness + honest counts) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-11 | Tampering | client-side delete-row mutating persisted data | accept | delete-row mutates ONLY client `rows` state; nothing is persisted until `confirmImport`, which re-reads the server-persisted `parsed_rows` (WR-01) — a deleted row is simply absent from the confirm payload, no integrity surface |
| T-13-12 | Repudiation / Integrity | summary mis-count after delete (deletes counted as duplicates) | mitigate | `descartadas`/`duplicadas` read from the server-provided summary, NOT the rows-length delta; deletes tracked separately so counts stay honest (RESEARCH Open Question 4) |
| T-13-13 | Info disclosure | review route rendering a silent empty grid on 0 rows | mitigate | the `Nenhuma transação reconhecida` Empty (text-present 0-rows) + the upstream image-only `{ error }` block (13-03) guarantee the user always sees the file was read (PDF-04) |
</threat_model>
