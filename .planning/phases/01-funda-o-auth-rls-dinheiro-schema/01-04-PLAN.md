<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local secret store (.env.local) → git | a committed `.env.local` would leak the project credentials |
| local migrations → remote project | a wrong `db push` could clobber pre-existing remote data |
| deployed bundle → public internet | the live client JS is world-readable; a secret here is fully exposed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-envleak (T1) | Information Disclosure | `.env.local` committed / secret in Vercel public env | mitigate | `.env.local` gitignored (Plan 01); secret (if any) is NOT `NEXT_PUBLIC_*`; deployed-bundle grep spot check in Task 3 (SEC-02) |
| T-1-clobber | Tampering | `db push` overwriting pre-existing remote tables | mitigate | `supabase db diff` reviewed before first push (Pitfall 4); migrations are idempotent `if not exists` |
| T-1-rls-remote (T2/T4) | Information Disclosure | RLS not actually enforced on the remote project | mitigate | dashboard check that tables are NOT "Unrestricted" + live signup shows only the new user's own categories |
| T-1-storage-remote (T3) | Information Disclosure | remote `statements` bucket public | mitigate | dashboard check that the bucket is PRIVATE before any upload flow lands (Phase 4) |
</threat_model>
