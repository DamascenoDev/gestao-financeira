<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| any authenticated user → Postgres | every row read/write crosses RLS; a missing/loose policy leaks another user's financial data with NO error (empty result) |
| any authenticated user → Storage | object reads/writes crossing the `{user_id}/` folder boundary |
| signup event → DB trigger | `SECURITY DEFINER` function runs with elevated rights; a wrong `search_path` is a privilege-escalation vector |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-rls (T2) | Information Disclosure | `categories`/`profiles` RLS — cross-user read leak; denial returns empty, not error → silent leak | mitigate | `enable row level security` + `(select auth.uid()) = user_id` USING `TO authenticated`; verified by `tests/rls-isolation.test.ts` SELECT-as-user-B = 0 rows |
| T-1-check (T4) | Tampering | RLS policies missing `WITH CHECK` → cross-user INSERT/UPDATE | mitigate | every policy carries `with check ((select auth.uid()) = user_id)`; isolation test exercises INSERT/UPDATE/DELETE across `user_id` and asserts failure |
| T-1-storage (T3) | Information Disclosure | `statements` bucket public or unscoped → enumerable financial files | mitigate | bucket created `public=false`; `storage.objects` policy scoped to `(storage.foldername(name))[1] = (select auth.uid())::text`; no upload/getPublicUrl path exists in Phase 1 |
| T-1-defdef | Elevation of Privilege | `handle_new_user` `SECURITY DEFINER` with mutable search_path | mitigate | function pins `set search_path = public`; only inserts the caller's own `new.id` rows |
</threat_model>
