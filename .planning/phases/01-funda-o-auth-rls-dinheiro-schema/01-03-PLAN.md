<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Server Action | untrusted FormData (email/password) crosses into server auth mutations |
| Server Component / middleware → session claims | a spoofable session check (`getSession`) would let an attacker forge auth |
| server module → client bundle | a secret-key import pulled into a client component leaks the key to the browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-secrets (T1) | Information Disclosure | secret key imported into a client component → shipped in bundle (SEC-02) | mitigate | client/server split uses only the publishable key; no `src/lib/supabase/`, `src/actions/`, or page imports the secret; `check-bundle-secrets.sh` + `bundle-secret-grep.test.ts` enforce green after build |
| T-1-spoof | Spoofing | session check via `getSession()` (cookie-spoofable) | mitigate | middleware + `(app)/layout.tsx` use `getClaims()` (validates JWT signature against published keys), never `getSession()`/`getUser()` for protection |
| T-1-input | Tampering / Input Validation | unvalidated email/password into auth actions | mitigate | Zod schema validates FormData at the Server Action boundary before any Supabase call (V5) |
| T-1-bypass | Elevation of Privilege | relying only on middleware for route protection | mitigate | defense-in-depth: `(app)/layout.tsx` re-checks `getClaims()` server-side and redirects |
</threat_model>
