# Phase 20 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope: INTENTIONALLY TINY.** Phase 20 is mostly backend (deterministic keyword-matching layer in the upload pipeline: memória → palavra-chave → IA). The ONLY new visible UI is **one new provenance pill labelled "palavra-chave"** in the import review grid, mirroring the EXISTING **"memória"** pill exactly. ZERO new tokens, palette, typography, spacing, shadcn components, or layout. Every value below is REUSE of the `base-nova` navy+gold system (Phase 7) and the provenance/affordance grammar locked in Phase 16. This contract records the exact tokens, casing, placement, and mutual-exclusivity so the executor adds the new branch with zero design ambiguity. The interaction is already LOCKED in `20-CONTEXT.md` (KW-02/KW-05): a keyword-matched row arrives PRE-FILLED (binding category) with the neutral "palavra-chave" pill, overwritable exactly like a memória row.

## Inheritance Statement

This phase inherits the ENTIRE established system. The 6 pillars below read: **"inherits existing system; one new neutral pill."** Nothing new is introduced except:
- **One provenance label** `"palavra-chave"` rendered in the SAME pill markup and SAME neutral token as the existing `"memória"` badge.
- **One new origin/source state** threaded through the client `ReviewRow.origin` union + the `OriginBadge` variant union + the `ClassificationSource` union (`'palavra-chave'`).

There are **two pill surfaces** in the review grid that today show "memória" and MUST both gain the parallel "palavra-chave" state:
1. `ProvenanceBadge` (in the **Categoria** cell, `import-review-table.tsx` ~line 131) — neutral pill, no icon.
2. `OriginBadge` (in the **Origem** column, `origin-badge.tsx`) — neutral pill WITH a `lucide-react` icon.

## ⚠️ Flag — exact match the new pill MUST honor (the "memória" treatment)

The keyword pill is **deterministic/owned, like memory** — NOT the gold/accent IA-suggestion treatment. It must visually read as a sibling of "memória", not of "IA". The two existing surfaces use DIFFERENT neutral tokens — match each per-surface:

| Surface | File / line | "memória" treatment to mirror | New "palavra-chave" treatment |
|---------|-------------|-------------------------------|-------------------------------|
| `ProvenanceBadge` (Categoria cell) | `import-review-table.tsx:131-141`, via `AffordancePill` | `AffordancePill` + `bg-secondary text-secondary-foreground`, **NO icon**, label literal `memória` (lowercase) | SAME `AffordancePill` + `bg-secondary text-secondary-foreground`, **NO icon**, label literal `palavra-chave` (lowercase) |
| `OriginBadge` (Origem column) | `origin-badge.tsx:22-26` (`memória` variant) | `bg-muted text-muted-foreground` + `Brain` icon (`size-3`), label `Memória` (Title Case) | New `'palavra-chave'` variant: `bg-muted text-muted-foreground` + a `lucide-react` icon (recommend `Tags` or `Tag`) (`size-3`), label `Palavra-chave` (Title Case) |

**Casing rule (LOCKED):** match the per-surface casing of the existing "memória" label — `ProvenanceBadge` uses lowercase (`memória` → `palavra-chave`); `OriginBadge.VARIANT[].label` uses Title Case (`Memória` → `Palavra-chave`). Do NOT introduce a third casing.

**Token rule (LOCKED):** neutral only — `bg-secondary/text-secondary-foreground` (ProvenanceBadge) and `bg-muted/text-muted-foreground` (OriginBadge), exactly as "memória". NEVER the gold `bg-primary/10 text-primary` IA treatment, NEVER `--consumption` amber (that is "não classificada"), NEVER `--destructive`.

**Icon rule:** `ProvenanceBadge` "memória" has NO icon → the keyword pill in that surface has NO icon. `OriginBadge` "memória" HAS the `Brain` icon → the keyword variant gets its OWN distinct neutral icon (`Tags`/`Tag` recommended; `Brain` stays reserved for memória so the two states remain distinguishable beyond the label). Per the established a11y rule, the text label is always present so color/icon is never the sole signal.

The full system is the EXISTING Phase 7 navy+gold OKLCH set in `globals.css`. This change adds ZERO new colors and uses ONLY the neutral chrome already carrying the "memória" pill.

| Role | Value | Usage |
|------|-------|-------|

| Secondary (30%) | `--secondary` `oklch(0.95 0.012 255)` / `--muted` `oklch(0.955 0.01 255)` | The NEUTRAL pill ground for **both** the "memória" pill and the NEW "palavra-chave" pill (ProvenanceBadge → `bg-secondary`; OriginBadge → `bg-muted`) |
| Accent (10%) | `--primary` (gold) `oklch(0.76 0.13 88)` / dark `oklch(0.82 0.13 88)` | RESERVED for the AI signal only (SuggestionSlot chip + "IA" badge). **NOT used by the keyword pill** — keyword is deterministic/owned like memory, so it is neutral, never gold. |
| Destructive | `--destructive` `oklch(0.58 0.22 27)` | NOT used in this phase. |

**Accent (gold `--primary`) reserved for, this phase:** nothing new. The keyword pill is explicitly NEUTRAL and must NEVER take the gold IA treatment. `--consumption` amber stays reserved for the "não classificada" state; the keyword pill must NOT use it (a keyword-matched row IS classified).

**Provenance token map (mutually exclusive per row — extended for Phase 20):**

| State | Condition (review-time) | ProvenanceBadge (Categoria cell) | OriginBadge (Origem column) |
|-------|--------------------------|----------------------------------|------------------------------|
| memória | `category_id` set AND `origin === 'memória'` | `bg-secondary text-secondary-foreground`, no icon, `memória` | `bg-muted text-muted-foreground`, `Brain`, `Memória` |
| **palavra-chave (NEW)** | `category_id` set AND `origin === 'palavra-chave'` | `bg-secondary text-secondary-foreground`, no icon, `palavra-chave` | `bg-muted text-muted-foreground`, `Tags`/`Tag`, `Palavra-chave` |
| manual | `category_id` set AND `origin === 'manual'` | none (no chip) | `bg-muted text-muted-foreground`, `Pencil`, `Manual` |
| IA (unapplied suggestion) | `category_id === null` AND `suggestion.categoryId !== null` | `bg-primary/10 text-primary`, `Sparkles`, `IA` | n/a (Origem shows "não classificada" until applied) |
| não classificada | `category_id === null` | none | `bg-consumption/10 text-consumption`, `TriangleAlert`, `Não classificada` |

**Mutual exclusivity (LOCKED):** a keyword-matched row has `category_id` set, so it can NEVER co-render with the "não classificada" or "IA" (unapplied) states — exactly the same `category_id !== null` gate that already keeps memória and IA from co-showing. memória PREVAILS over palavra-chave at classify time (KW-03), so a single row is never both: the `origin` field carries exactly ONE value.

**Color never the sole signal:** the keyword pill always renders its text label (`palavra-chave` / `Palavra-chave`), per the inherited a11y rule. The OriginBadge variant also carries a distinct icon.

## Interaction & Layout Contract

**No layout change.** The keyword pill occupies the SAME reserved slots the memória pill uses today:
- In the **Categoria** cell (`import-review-table.tsx` `flex flex-wrap items-center gap-1` affordance row), `ProvenanceBadge` renders the neutral keyword pill in place of the memória pill when `origin === 'palavra-chave'`. The cell does not reflow.
- In the **Origem** column, `OriginBadge` renders the keyword variant. Both desktop table and mobile card paths read the SAME `OriginBadge` (the mobile card at `import-review-table.tsx:814` already mirrors the desktop `origin` cell), so the new variant appears in both with no extra work beyond the variant-map entry.

**Origin derivation (Claude's discretion per CONTEXT.md):** the client `ReviewRow.origin` derives `'palavra-chave'` from the parsed row's `classification_source`. The `ReviewRow.origin` union (currently `'memória' | 'manual' | 'não classificada'`) and the `OriginVariant` union (currently `'memória' | 'manual' | 'não classificada' | 'sugerida'`) must both gain `'palavra-chave'`. The `OriginBadge` call sites already pass `'não classificada'` when `category_id === null`, so a keyword row (`category_id` set) flows its `origin` straight through.

**Overwrite mechanics (LOCKED):** the Categoria `Select` is unchanged. Picking a new category runs the existing `classifyRow` → sets `origin: 'manual'` → the keyword pill disappears and the Origem flips to "Manual", identical to overwriting a memória row. No new dialog, chip, or button.

**No Phase-20 leakage into UI:** NO match-preview, NO "this keyword matched N transactions", NO tx-count, NO auto-suggestion UI — all out of scope (CONTEXT.md). The only visible delta is the pill label.

## Accessibility Basics

- **Color never the sole signal:** the keyword pill carries its text label in both surfaces; the OriginBadge variant also carries a distinct icon (`Tags`/`Tag`, not `Brain`) so memória and palavra-chave are distinguishable beyond hue.
- **Distinct from neighbors:** neutral ground matches memória (intended — same "deterministic/owned" family) but the label text differs, and the icon differs in the OriginBadge surface; it is never confusable with the gold IA pill or the amber "não classificada" pill.
- **No new interactive element:** the pill is a non-interactive `<span>` (same as memória); overwrite uses the existing keyboard-operable `Select`. No new focus/tab-order concerns.
- Honors `prefers-reduced-motion` (inherited — pill is static, no decorative motion).
