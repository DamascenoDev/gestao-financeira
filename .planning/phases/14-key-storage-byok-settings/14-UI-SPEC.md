## Color

Consume `src/app/globals.css` OKLCH tokens (light + dark already defined). No new hues.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (oklch 0.985 0.004 255 light / 0.2 0.03 258 dark) | Page canvas |
| Secondary (30%) | `--card` + `--muted`/`--secondary` | The single settings Card; muted help text via `--muted-foreground` |
| Accent (10%) | `--primary` (muted metallic gold) | Reserved list below |
| Destructive | `--destructive` (oklch 0.58 0.22 27 light) | "Remover chave" affordance + its AlertDialog confirm only |

**Accent (gold `--primary`) reserved for:**
1. The primary submit button "Salvar" (default `Button` variant).
2. The input focus ring (`--ring`, gold) — inherited from primitives, no extra work.
3. The "Chave configurada ✓" success badge MAY use the `income`/green-semantic or default badge — see state table. Gold is NOT spent on the badge; gold stays on the single primary CTA.

Gold is NOT used for: the "Testar conexão" button (use `variant="outline"` or `secondary` — it is a secondary affordance), body text, borders, or the status badge background.

**Status / semantic color mapping for this phase:**
- No-key badge → `Badge variant="secondary"` (neutral "Nenhuma chave configurada").
- Key-configured badge → `Badge variant="outline"` or `secondary` with a check glyph; do NOT use `--destructive`. Tone = calm/positive, not alarmed.
- Test-success inline → `text-income` (the established green-semantic) or muted-positive; concise "Conexão ok ✓".
- Test-error inline + save-error → `text-destructive text-sm` (matches the conta `settingsError` pattern). Friendly message, never a stack/secret.
- Remove-key danger affordance → `--destructive` (button + AlertDialog), mirroring the `AccountDeleteZone` danger-zone discipline already in the repo.

## Interaction States (the security-critical contract)

The API key is **write-only**. These distinct, honest states are required (CONTEXT.md security invariant — checker must confirm all are visually represented):

| State | Trigger | Visual contract |
|-------|---------|-----------------|
| no-key (first run) | `has_key === false` | Gemini pré-selecionado · key input EMPTY (never pre-filled) · placeholder shown · badge `Nenhuma chave configurada` (secondary) · "Testar conexão" disabled until a key exists OR is typed |
| key-configured | `has_key === true` | badge `Chave configurada ✓` · key input EMPTY with key-configured help text (NEVER echoes the stored key — no `value` pre-fill, no masked dots representing the real key) · "Testar conexão" enabled · "Remover chave" visible |
| typing | user enters key | input is `type="password"` (masked dots are the user's own typed chars, not the stored secret) · Salvar enabled |
| saving | `Salvar` clicked | button disabled, label `Salvando…` (useTransition) |
| save-success | action returns `{ ok }` | `toast.success('Configurações de IA salvas')` · input clears · badge flips to `Chave configurada ✓` |
| save-error | action returns `{ error }` | `toast.error` with friendly copy; never leak stack/secret |
| testing | `Testar conexão` clicked | button disabled, label `Testando…`; inline result region in `aria-live="polite"` |
| test-success | `{ ok }` | inline `Conexão ok — sua chave está funcionando.` in `text-income`/positive |
| test-error | `{ error }` | inline friendly mapped message (invalid / no-credits / network) in `text-destructive`; never a raw provider error |
| removing → removed | `Remover chave` → AlertDialog confirm | AlertDialog (focus Cancelar) → on confirm: `toast` removed copy · badge back to `Nenhuma chave configurada` · app returns to pre-IA state without breaking (BYOK-05) |

**Hard visual rule for the checker:** there must be NO code path where a stored key (any `sk-…` / `AIza…`) is rendered as an input `value`, a masked-but-real placeholder, or any text node. The form projects only `has_key` (boolean) + `provider`. Save persists; testing is a separate, optional affordance (a green test is NOT required to save).

**Accessibility:** key input `type="password"` + `autoComplete="off"`; status badge and inline test result are `aria-live="polite"`; AlertDialog initial focus on the non-destructive button (mirror `AccountDeleteZone`); 44px minimum touch target on all buttons (default `Button` height already satisfies this).
