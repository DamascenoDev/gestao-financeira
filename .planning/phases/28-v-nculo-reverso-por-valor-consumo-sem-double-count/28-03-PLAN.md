<objective>
Gravar o **vínculo** no `confirmImport` — o seam exato logo após o insert das transações (D-08).
Para cada linha com vínculo confirmado no payload:
- **à-vista** → `update abastecimentos set transaction_id = <txId>` no abastecimento escolhido +
  sync de `carro_id` na tx (espelha `abastecimentos.ts` L158-166);
- **parcelado** → `insert abastecimento_parcelas` (`user_id`, `abastecimento_id`, `transaction_id`,
  `parcela_num`), com `parcela_num` = já-na-junção + atribuídas-nesta-confirmação + 1.

Três proteções obrigatórias, espelhando os padrões já no arquivo:
1. **IDOR re-derive** do `abastecimentoId` (D-08) via `assertOwnedAbastecimento` tri-state ANTES
   de qualquer write — molde: o IDOR de `carro_id` em L728-746. FKs não são RLS-aware.
2. **D-09**: a tx dedupe-skipped (fora do `insertedByKey`) com vínculo confirmado é resolvida por
   um lookup batched `.in('dedupe_key', [...])` e ainda recebe o vínculo.
3. **Defense-in-depth de duplo-link**: os unique indexes da P26 são o backstop — um 23505 vira
   already-linked, nunca um 500; falha de vínculo APÓS o insert da tx é surface-but-keep (espelha
   LEARN L985-989).

Mais uma suíte DB-integration que prova as cinco invariantes.

Purpose: Entregar CAR-10 (vínculo gravado no confirm, sem auto-commit antes) + a metade de
gravação de CAR-11 (parcela única + sem double-count via parcela_num + unique indexes), com o
IDOR como gate de segurança L1 bloqueante.
Output: `confirmImport` estendido + `tests/import-abastecimento-link.test.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente → confirmImport (Server Action) | O payload do cliente carrega `abastecimentoId` + `abastecimentoKind` + `parcelaNum` (a ESCOLHA do vínculo). Este é o boundary não-confiável crítico desta fase. |
| confirmImport → Postgres (FKs) | As FKs `abastecimentos.transaction_id` / `abastecimento_parcelas.abastecimento_id` NÃO são RLS-aware — uma FK forjada para a linha de outro usuário satisfaz o constraint globalmente. |
