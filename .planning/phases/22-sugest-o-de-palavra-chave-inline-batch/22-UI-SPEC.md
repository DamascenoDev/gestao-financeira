## Interaction & Layout Contract

### Inline (KW-07) â€” import review grid

**Gating (LOCKED).** The "+ palavra-chave" affordance renders ONLY when the row's classification origin is `manual` (the user picked/overrode the category by hand). Rows classified by `memĂ³ria` / `palavra-chave` / `IA` do NOT show it â€” the descriptor is already learned/covered, so offering a keyword there is noise. Read the same `origin` field the grid already tracks (set by `classifyRow` at ~344â€“359).

**Placement.** The affordance lives on the chip's EXISTING reserved row inside `InlineReviewCategoryCell` (the `flex flex-wrap items-center gap-1` div at ~977, alongside `ProvenanceBadge`/`ConfidenceTag`). It adds NO new column and does NOT reflow the Categoria column (reuse `SuggestionSlot`'s `min-h-5` pill geometry).

**Flow.**
1. User manually picks/overrides a category on a row â†’ `origin` becomes `manual` â†’ the "+ palavra-chave" pill appears.
2. Click the pill â†’ a small `Popover` opens, anchored to the pill, containing one `Field`: `FieldLabel` "Palavra-chave" + `Input` prefilled with the normalized `descriptor_norm` (editable, `maxLength={60}`) + the optional helper line + a `flex gap-2` footer with "Cancelar" (outline) and "Salvar" (primary).
3. On "Salvar" â†’ `startTransition(async () => addKeyword(rowCategoryId, value))` â€” the keyword is created in the category the user JUST picked for that row (no second question). This write is DECOUPLED from "Confirmar importaĂ§Ă£o": it writes ONLY `category_keywords`, never `transactions`/`merchant_patterns`.
4. On `{ok:true}` â†’ close popover, toast success, flip the inline control to disabled "criada âœ“". On `{duplicate}` â†’ toast.info "jĂ¡ cadastrada", still flip to "criada âœ“" (it exists now). On `{error}` â†’ keep the popover open, show inline `FieldError`, do not flip.

**Idempotence.** Once "criada âœ“", the control stays disabled for that row for the rest of the review session (prevents a double-create on the same descriptor). It does not need to survive a page reload (session-scoped is sufficient).

### Batch (KW-08) â€” global dialog from `/categorias`

**Entry point.** A new **"Sugerir palavras-chave"** button in the `/categorias` header row (beside "Nova categoria"). It toggles a controlled-open global `Dialog` (candidates cross categories â†’ global, not per-row). Whether candidates are computed in the RSC fetch or in a server action on dialog open is Claude's discretion (CONTEXT.md); the UI contract is the same either way.

**Candidate source & ordering (LOCKED).** Candidates = confirmed `merchant_patterns` (`descriptor_norm` â†’ `category_id`), EXCLUDING any descriptor already covered by an existing keyword (server-side `matchKeyword` filter), ordered by `hit_count` desc (highest-value first). NOT mined from raw `transactions`.

**Dialog anatomy (top â†’ bottom), composing reused primitives:**
1. `DialogHeader` â†’ `DialogTitle` ("SugestĂµes de palavras-chave") + `DialogDescription`.
2. **Candidate list** â€” a scrollable `flex flex-col gap-2` region (`max-h` + `overflow-y-auto`). Optional list header with a "Selecionar todas" checkbox. Each candidate row is `flex items-center gap-2`:
   - a `Checkbox` (multi-select),
   - the editable term `Input` (prefilled normalized `descriptor_norm`),
   - the editable category control (chip + `Select` of the user's categories, prefilled from the pattern),
   - the "{N} usos" hint (`text-muted-foreground tabular-nums`),
   - a neutral muted discard `X` button (`aria-label` above) that removes the row from the list (session-only, no write).
   - When the candidate list is empty, render the `Empty` primitive (heading + body copy above) INSTEAD of the list.
3. `DialogFooter` â†’ the primary **"Aprovar selecionadas ({N})"** button (disabled at 0 selected) + a `DialogClose` "Fechar" (outline).

**Bulk approve.** Click "Aprovar selecionadas" â†’ `startTransition` â†’ create the selected candidates as `category_keywords` (each via `addKeyword`, or one batch action `addKeywords`/`approveKeywordSuggestions` â€” Claude's discretion, same conventions as `category-keywords.ts`: Zod boundary, `getClaims` owner-gate, dedupe, single `revalidatePath('/categorias')`). On success â†’ toast the created/skipped count, remove the approved rows from the list, keep the dialog open so the user can continue (or close via "Fechar"). On error â†’ toast.error, leave selections intact.

**Discard.** Click a candidate's `X` â†’ the row leaves the list immediately (local state), NO server write, NO confirm. It may reappear on a future scan (accepted).

**Edit-before-approve.** Both the term and the category of each candidate are editable BEFORE approval. Edited terms pass through `normalizeKeyword`; an empty/`*`-only normalized term surfaces the same inline validation messages as the inline popover (block that one row's approval, not the whole batch).

**Keyboard & focus (LOCKED, inherited Base UI defaults):**
- Inline popover: opens on click, focus moves to the term `Input`; Enter submits "Salvar"; Esc closes without writing; focus returns to the affordance pill on close; focus trapped while open.
- Batch dialog: on open focus lands in the dialog (first interactive control); Esc closes; focus returns to the "Sugerir palavras-chave" trigger; focus trapped while open. Each checkbox, input, select, discard `X`, and the approve/close buttons are keyboard-operable in DOM order (checkbox â†’ term â†’ category â†’ discard, per row; then footer).

**Loading state.** While an action is in flight (`isPending`): the inline "Salvar" / batch "Aprovar selecionadas" button is disabled and shows its pending label ("Salvandoâ€¦" / "Aprovandoâ€¦"). Do NOT block the whole dialog or show a spinner overlay â€” actions are fast and the disabled button is sufficient (matches `category-keywords-dialog.tsx`).

**No scope leakage.** No change to the match algorithm, the upload classification pipeline, procedĂªncia persistence, or any auto-creation. Nothing is written to `transactions`/`merchant_patterns`. No new table/column (discard is session-only).

## Accessibility Basics

- **Color never the sole signal:** the inline affordance carries the "+ palavra-chave" / "criada âœ“" text + glyph + `aria-label`; candidate selection is a real `Checkbox` (`aria-checked`, not hue); the category chip carries its name; the usage hint is a literal "{N} usos"; the discard control carries `aria-label="Descartar sugestĂ£o {keyword}"`. Nothing relies on hue alone (inherited Phase 7 rule).
- **Real buttons / inputs:** the inline affordance, popover "Salvar"/"Cancelar", every checkbox, discard `X`, and approve/close buttons are real focusable controls (not clickable spans), Enter/Space-activatable, each with a descriptive label.
- **Labels:** the inline popover `Input` has an associated `FieldLabel` (`htmlFor`/`id` via `useId`); each candidate term input and category select are labeled (visible label or `aria-label` keyed to the candidate descriptor so screen-reader users can disambiguate rows).
- **Focus:** visible gold ring (`--ring`) on all controls; focus moves into the popover/dialog on open and returns to the trigger on close; focus trapped while open (Base UI default â€” do not override).
- **Live feedback:** success/duplicate/error use `sonner` toasts (themed + `aria-live` by the toaster); inline validation uses `FieldError` associated to the popover input for SR announcement.
- **Touch targets:** the inline `size-3` glyph sits inside the `min-h-5` pill; checkboxes and the discard `X` meet the inherited mobile touch guidance.
- Honors `prefers-reduced-motion` (inherited â€” popover/dialog and list add/remove are opacity/layout only, no decorative motion).
- **Inline gating is also an a11y win:** showing the affordance only on `manual` rows keeps the grid quiet and avoids a phantom control on already-classified rows.
