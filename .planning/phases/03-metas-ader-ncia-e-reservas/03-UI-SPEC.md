# Phase 3 — UI Design Contract

> Visual and interaction contract for the "visão de metas" — the core-value screen. Scope: Dashboard de aderência (mensal + YTD), Definir metas (% por categoria + direção), Reservas (sinking funds com saldo derivado), e o sub-fluxo "qual reserva?" no lançamento de transação.
>
> **This phase INHERITS the app-wide design system established in Phase 2** (`02-UI-SPEC.md`): teal `--primary` accent (reserved), OKLCH semantic money tokens (`--income`/`--expense`/`--allocation`/`--consumption`), `--font-mono` tabular numbers, dense layout, `base-nova` shadcn, pt-BR copy, money only through `src/lib/money.ts`. This spec does **NOT** redefine that system — it extends it for the new screens. Where this doc is silent, the Phase 2 contract governs.
>
> Generated by gsd-ui-researcher. Stack locked: Next.js App Router, TS strict, Tailwind v4 (`@theme`), shadcn `base-nova`, lucide, pt-BR. Adherence/balances computed in SQL views (`security_invoker=true`), never in the app.

## Semantic Color — Direction-Aware Adherence (the one real extension)

No new tokens. Adherence STATUS is computed by the view (`gasto/alocado ÷ receita líquida` vs `meta %`) and the **direction** (`teto` for consumo, `alvo` for alocação) flips which side is good. The UI maps status → existing token:

| Direction | Condition | Status | Fill token | Text/badge token | Meaning |
|-----------|-----------|--------|------------|------------------|---------|
| **teto** (consumo, não exceder) | `< 80%` da meta | ok | `--consumption` (amber, calm) | `text-muted-foreground` | dentro do limite |
| **teto** | `≥ 80%` e `< 100%` | aproximando | `--consumption` (amber, fuller) | amber + alert glyph | chegando no teto |
| **teto** | `≥ 100%` | estourou | `--destructive` (red) | `text-destructive` + alert | passou do teto |
| **alvo** (alocação, atingir) | `< 80%` da meta | abaixo | `--muted-foreground` track-fill | `text-muted-foreground` | ainda longe do alvo |
| **alvo** | `≥ 80%` e `< 100%` | quase lá | `--allocation` (indigo) | indigo | perto do alvo |
| **alvo** | `≥ 100%` | atingido | `--income` (green) | `text-income` + check | meta batida |

**Rules carried over from Phase 2 that this respects:**
- A gasto is **never painted red just for being a gasto** — red appears ONLY when a `teto` is `≥100%` (over-limit). This is exactly the "Phase 3 concern" the Phase 2 spec deferred.
- Green (`--income`) means "good outcome", here = `alvo` atingido. It is consistent with income = money in.
- The neutral chrome stays grayscale; color appears only on the bar fill, the % figure, and the alert glyph — never on card backgrounds.
- Color is never the sole signal: every state also shows a text status label (`No limite` / `Aproximando` / `Estourou` / `Atingido`) and an icon, so it survives color-blindness and grayscale.

`@theme` already exposes `bg-income`/`bg-consumption`/`bg-allocation`/`bg-destructive` — no globals.css change needed for color.

### 1. Dashboard de metas / aderência (`(app)/dashboard`) — core-value screen

Replaces the Phase-1 placeholder. The single most important screen of the product: "o quanto estou aderente às metas".

- **Header:** h1 "Metas e aderência" + the shell `MonthSelector`. Primary CTA **"Definir metas"** (teal) → routes to / opens the Definir metas surface (§2).
- **Period toggle (`tabs`):** **Mensal** (default, uses `?mes=`) and **Anual (YTD)**. Both derive from the **same ledger** — the Anual view is the same `%` meta applied to the **accumulated** receita líquida of the year vs accumulated gasto/alocado. They MUST read as visibly consistent (same row layout, same color logic, same category order). Label the YTD view explicitly: "Acumulado de 2026 (jan–jun)" so the user sees the window.
- **Summary strip (`AdherenceSummaryStrip`):** período label · receita líquida do período (hero, 28px mono, `text-income`) · a terse count e.g. "2 categorias estouradas · 1 meta atingida".
- **Per-category list:** one `AdherenceRow` per category that has a meta, grouped/ordered consumo-then-alocação (or by severity — executor's choice, but stable across Mensal/Anual). Each row carries the direction-aware bar, realized R$ + %, meta %, and status.
- **Alocação grouping note (RSV-03, locked):** categorias de **alocação** (Investimentos + Reserva) **somam JUNTAS** na meta de investimento. Render the combined alocação meta as one row (or a parent row with a small "inclui aportes de reserva" note) — aportes de reserva contam aqui como alocação de investimento e **nunca** como gasto de consumo. Make this legible: a `tooltip`/helper "Inclui aportes de reserva".
- **Alert affordance (BUD-04):** at `≥80%` show an amber **"Aproximando"** badge + `triangle-alert` glyph on the row; at `≥100%` show the terminal state — for `teto`: red **"Estourou"** + `octagon-alert`; for `alvo`: green **"Atingido"** + `check`. Alerts are inline on the row AND surfaced in the summary count. No toast/notification system in v1 — the dashboard IS the alert surface.
- **States:**
  - **Empty (no metas defined yet):** `empty` primitive — heading **"Você ainda não definiu metas"** · body "Defina uma meta em % da sua receita líquida para cada categoria e acompanhe sua aderência aqui." · CTA **"Definir metas"**. This is the expected first-run state.
  - **Empty (metas defined, no receita this period):** info row "Sem receita líquida em {período} — as metas em % não podem ser calculadas. Cadastre uma receita." with a link to Receitas. (Division-by-zero guard surfaced as copy, never `NaN%`.)
  - **Loading:** `skeleton` rows (keep header + period tabs visible).
  - **Error:** inline `text-destructive` "Não foi possível carregar a aderência. Tente recarregar a página."

### 2. Definir metas (`MetaDialog`, surfaced from dashboard and/or Categorias)

Set a `%` target per category with a direction. Inline-per-category is the primary pattern; a list view on the dashboard is acceptable.

- **Trigger:** "Definir metas" CTA (dashboard) opens a surface listing the user's categories, each with: nome (`CategoryBadge`), a `%` input, a direction `switch` (**Teto** ↔ **Alvo**), and a **live R$ preview** (`% × receita líquida do mês` via `formatCents`, recomputed as the user types). One meta per category (BUD-01).
- **Direction default:** derived from `kind` — consumo → **Teto** (não exceder), alocação → **Alvo** (atingir). Editable. Show the meaning inline: under a Teto, helper "Não exceder {R$}"; under an Alvo, "Atingir {R$}".
- **% input:** numeric, 0–100, one decimal allowed; reject `>100` per single category? — allow, but soft-warn if the **sum of teto metas > 100%** ("Suas metas de teto somam mais de 100% da receita"). Mono display.
- **Validation/ownership:** every `category_id` from the client is validated server-side by owner before write (carried-over IDOR lesson — RLS does not close a client-supplied FK to another user's row).
- **Save:** `sonner` success "Meta de {categoria} salva". Clearing the % removes the meta (with a subtle confirm or just empties it).
- **Empty:** if no categories exist (unlikely — seed exists), link to Categorias.

### 3. Reservas (`(app)/reservas`) — sinking funds

List of named reservas with **derived** balance (Σ entradas − Σ saídas, never a stored column), optional alvo, ledger, and validated saídas.

- **Header:** h1 "Reservas" + primary CTA **"Nova reserva"** (`ReservaForm` dialog).
- **List:** responsive grid of `ReservaCard`. Each: nome · **saldo** (mono hero, `AmountCell` neutral) · `ReservaProgress` **only if `alvo_cents` set** (bar + "{saldo} de {alvo} · {%}") · actions (`dropdown-menu`): **Registrar saída** · **Ver extrato** · **Editar** · **Excluir**.
- **Progress rule (RSV-01/RSV-05):** the bar appears **only when há alvo**. No alvo → no bar at all (not an empty/0% bar). Bar fill `--primary`, switching to `--income` at ≥100% ("Alvo atingido" badge).
- **Saldo:** always derived via view; the UI never computes or mutates it client-side.
- **Reserva detail / ledger:** clicking "Ver extrato" opens the per-reserva ledger (a detail panel/route or a `sheet`/`dialog`): header with nome + saldo + alvo progress, then `ReservaLedgerTable` (entradas/saídas history, newest first, with link back to the source transação when `transaction_id` set). Primary action here: **"Registrar saída"**.
- **Registrar saída (`SaidaForm`, RSV-04):** dialog — valor (`MoneyInput`), data, descrição (optional). **Validated `valor ≤ saldo atual`** both client-side (block submit, inline error) and server-side (authoritative). A saída reduces saldo, is a movimento da reserva, and **does NOT affect gasto metas**. Success toast "Saída registrada".
- **Create/edit (`ReservaForm`):** nome (required, `input`), alvo (optional `MoneyInput`). Editing alvo to empty removes the progress bar.
- **Delete:** `alert-dialog`. Reserva com ledger → warn that history will be removed (or block + offer archive — executor's choice, but never silently orphan ledger rows). Copy in §Copywriting.
- **States:**
  - **Empty (no reservas):** `empty` — heading **"Nenhuma reserva ainda"** · body "Crie uma reserva (ex.: Apê, Carro) para guardar dinheiro por objetivo. Lançamentos na categoria Reserva entram aqui." · CTA **"Nova reserva"**.
  - **Empty ledger (reserva exists, no movimentos):** "Nenhum movimento nesta reserva ainda. Um lançamento na categoria Reserva cria a primeira entrada."
  - **Loading:** `skeleton` cards / ledger rows.
  - **Error:** inline `text-destructive` "Não foi possível carregar as reservas. Tente recarregar a página."

### 4. "Qual reserva?" sub-flow (integrates into the Phase 2 transação flow)

When a transaction is classified into the seed **"Reserva"** category, the flow must ask **which** reserva the aporte goes to, then create an `in` (entrada) in that reserva's ledger linked to the transação (RSV-02).

- **Where it fires:** in `transacao-form.tsx` (Novo lançamento / edit) **and** on inline category re-tag in the Extrato when the chosen category is "Reserva". The sub-question is a **conditional field inside the same dialog**, not a second modal — when `categoria === "Reserva"`, a `ReservaPicker` field appears below the categoria select (progressive disclosure, no extra modal hop). On inline Extrato re-tag (where there's no open dialog), selecting "Reserva" opens a small focused `dialog` containing just the `ReservaPicker`.
- **`ReservaPicker` contract:** a `select` of the user's reservas (by nome), required when categoria is Reserva. Includes an inline **"+ Nova reserva"** option/affordance that opens `ReservaForm` without losing the transação in progress (return to the picker with the new reserva pre-selected). Label: **"Qual reserva?"**, helper "Este lançamento será registrado como aporte nesta reserva."
- **Validation/ownership:** the chosen `reserva_id` (client-supplied) is validated server-side by owner before the ledger entry is written (IDOR lesson). If no reserva exists yet, the picker shows an inline empty state with the "+ Nova reserva" affordance and blocks submit until one is chosen/created.
- **Accounting (RSV-03, locked):** the resulting entry is an **alocação de investimento** — it contributes to the Investimentos meta and **never** to a consumo meta. No separate "gasto" is recorded for an aporte.
- **Edit/undo:** if a transação is later re-classified **away** from Reserva, its linked ledger entry must be removed (saldo re-derives). Surface this honestly — no silent orphan. (Behavior is a data concern; the UI just reflects the re-derived saldo.)
- **Copy:** picker title "Qual reserva?"; submit keeps the transação's existing "Adicionar"/"Salvar" button (no separate confirm).

## Money & Number Formatting

Inherited from Phase 2, with additions:

- **Money:** every value via `formatCents` (pt-BR `R$ 1.234,56`); every input via `parseBRLToCents` (throws → `FieldError` "Valor monetário inválido."). Mono + `tabular-nums`.
- **Percentages:** rendered mono, `tabular-nums`, `maximumFractionDigits: 1` → `72,5%`, `100%`, `0%`. Never show `NaN%`/`Infinity%` — guard division by zero with the "sem receita no período" copy.
- **Saldo / saída:** saldo derived (Σ in − Σ out); saída validated `≤ saldo`. Entrada renders `+ R$` green (`--income`), saída renders `− R$` neutral.
- **Dates:** `dd/MM/yyyy` (ledger uses `dd/MM`); period boundaries via `date-fns-tz` pinned to `America/Sao_Paulo`. YTD window = `startOfYear`..now in SP.
- **Adherence math:** computed in SQL views (`security_invoker=true`), never in the app — the UI only formats the view's output.

## Accessibility Basics

Inherited from Phase 2, plus:

- **Color never sole signal:** every adherence state pairs color with a text status label (`No limite`/`Aproximando`/`Estourou`/`Atingido`) and a lucide glyph — survives grayscale and color-blindness.
- **Bars:** `AdherenceBar` and reserva `progress` expose `role="progressbar"` + `aria-valuenow`/`aria-valuemin`/`aria-valuemax` + `aria-valuetext` (e.g. "Alimentação: 85% da meta de 30%, aproximando do teto").
- **Period tabs:** standard `tabs` keyboard semantics; selected period announced.
- **`ReservaPicker`:** required-field semantics with an accessible name "Qual reserva?"; the conditional reveal is announced (not a silent DOM swap).
- **Saída validation:** the `≤ saldo` error is a `FieldError` (aria-invalid + associated text), not just a disabled button.
- **Destructive confirms:** `alert-dialog`, focus-trapped, escape-cancellable.
- **Contrast:** direction-aware fills tuned ≥3:1 as non-text indicators; status text ≥4.5:1 on its surface in light and dark (tokens already tuned in Phase 2).
