# Phase 19 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** SMALL, additive UI on the EXISTING `/categorias` screen. ZERO new tokens, palette, typography, or shadcn components. One menu item enters `src/components/category-row-actions.tsx` and one new client Dialog (mirroring `categoria-form.tsx`) is added. All values below are REUSE of the established `base-nova` navy+gold system locked in Phase 7 (`07-UI-SPEC.md`) and the chip/affordance grammar refined in Phase 16 (`16-UI-SPEC.md`). This contract records the exact tokens, copy, states, a11y and keyboard behavior so the planner/executor have zero ad-hoc styling decisions. The interaction itself is already LOCKED in `19-CONTEXT.md` — this doc turns those decisions into a precise visual contract. **Reused components (do NOT recreate):**
- `src/components/category-row-actions.tsx` — per-row `DropdownMenu`. The new **"Palavras-chave (N)"** item is inserted here as a third `DropdownMenuItem`, BETWEEN "Editar" and the destructive "Excluir" (so the destructive action stays visually last). It toggles the new dialog's `open` state via local React state, EXACTLY like `editOpen`/`deleteOpen`.
- `src/components/categoria-form.tsx` — canonical Dialog pattern: controlled-or-uncontrolled `open`, `useTransition` (`isPending`), `toast` (sonner), `DialogHeader`/`DialogTitle`/`DialogDescription`/`DialogFooter`/`DialogClose`. The new keyword dialog MIRRORS this skeleton.
- `src/components/ui/badge.tsx` — shadcn `Badge`. Keyword chips use `variant="secondary"` with `className="gap-1"` (the exact removable-chip recipe already used in `category-filter.tsx:109`).
- `src/components/category-filter.tsx` (lines 106-122) — the precedent removable-chip: `<Badge variant="secondary" className="gap-1">{label}<button aria-label="Remover …"><X className="size-3" /></button></Badge>`. The keyword chip COPIES this markup (label = the normalized keyword; no `CategoryDot` needed — keywords have no color).
- `src/components/ui/field.tsx` (`Field`/`FieldLabel`/`FieldError`) — the add-keyword input/label/inline-error follow the `categoria-form.tsx` Field usage.
- `src/components/ui/empty.tsx` (`Empty`/`EmptyHeader`/`EmptyTitle`/`EmptyDescription`) — the in-dialog empty state when a category has no keywords.
- `src/app/(app)/categorias/page.tsx` — the RSC table. It gains a server fetch of keywords (list + per-category count) passed into `CategoryRowActions` so the menu label can show `(N)` and the dialog opens pre-populated.

**New components:** ONE new client component, `src/components/category-keywords-dialog.tsx` (exact filename at Claude's discretion per CONTEXT.md). No new shadcn primitives, no new tokens, no third-party registry. It composes only the reused primitives above.

- The chips container is `flex flex-wrap gap-2` so a long keyword list wraps inside the dialog and never forces horizontal scroll.

## Copywriting Contract

All copy pt-BR, calm/direct/second-person, no exclamation — matching the inherited `/categorias` and dialog voice.

| Element | Copy |
|---------|------|
| Menu item (in `CategoryRowActions`) | **"Palavras-chave"** when N=0, **"Palavras-chave ({N})"** when N≥1 — N from the RSC server fetch. Sits between "Editar" and "Excluir". |
| Dialog title | **"Palavras-chave — {Categoria}"** (interpolate the category name) |
| Dialog description | **"Adicione termos que identificam gastos desta categoria. Eles serão usados para classificar automaticamente futuras faturas."** (states purpose without promising the Phase-20 matching UI) |
| Input label | **"Nova palavra-chave"** |
| Input placeholder | **"Ex.: uber"** (lowercase — keywords are stored normalized) |
| Primary CTA (add) | **"Adicionar"** (verb; `Button` type=submit; pending → **"Adicionando…"**) |
| Chip remove button `aria-label` | **"Remover palavra-chave {keyword}"** |
| Success toast — add | **"\"{keyword}\" adicionada."** — `{keyword}` is the NORMALIZED value (matches the chip the server persists). _Ratified 2026-06-19 from the WR-01 code-review fix: echoing the normalized term keeps the toast consistent with the chip and the duplicate toast, vs the original generic "Palavra-chave adicionada." which could mismatch what was stored._ |
| Success toast — remove | **"Palavra-chave removida."** |
| Duplicate (friendly, NOT an error) | **toast.info** — **"\"{keyword}\" já está cadastrada nesta categoria."** (no inline FieldError, no red state — it is a benign no-op) |
| Inline validation error (empty after trim) | **"Informe uma palavra-chave."** (shown as `FieldError`, mirrors `categoria-form.tsx` "Informe o nome") |
| Inline validation error (too long) | **"Use até 60 caracteres."** (rely on `maxLength={60}` on the Input to prevent it; the message is the server-side fallback) |
| Error state (server action failure) | **toast.error** with the action's returned message; fallback **"Não foi possível salvar. Tente novamente."** |
| Empty state heading (no keywords) | **"Nenhuma palavra-chave"** |
| Empty state body | **"Adicione um termo abaixo (ex.: \"uber\") para começar a classificar esta categoria automaticamente."** |

**Normalization display rule (LOCKED from CONTEXT.md):** keywords are stored NORMALIZED via `normalizeDescriptor` (`src/lib/normalize.ts`) — the same function that produces `descriptor_norm`, so Phase 20's substring match is apples-to-apples. The chip and any echo display the **normalized value** (lowercase, accents stripped, punctuation collapsed), NOT the raw input. If normalization changes the input, the persisted/displayed chip is the normalized form (no separate "you typed X, saved Y" message — keep it calm). If `normalizeDescriptor(input) === ''` (empty after normalization), treat as the empty-validation error "Informe uma palavra-chave."

**No destructive confirmation dialog:** removing a keyword is immediate (per CONTEXT.md "persistência imediata por ação") and reversible (re-add), so it does NOT get an `alert-dialog` confirm. The `X` removes on click → server action → toast → revalidate. This is consistent with the single-field `setKind`/`setColor` toggles that also persist without confirmation. (The only destructive-confirm on this screen remains category deletion, untouched here.)

## Interaction & Layout Contract

**Entry point.** In `CategoryRowActions`, add a third `DropdownMenuItem` ("Palavras-chave (N)") and a third controlled-open state (e.g. `keywordsOpen`). Render `<CategoryKeywordsDialog open={keywordsOpen} onOpenChange={setKeywordsOpen} category={…} keywords={…} />` alongside the existing edit/delete dialogs (controlled, no inline trigger — same pattern as the edit dialog).

**Dialog anatomy (top → bottom), composing reused primitives:**
1. `DialogHeader` → `DialogTitle` ("Palavras-chave — {Categoria}") + `DialogDescription`.
2. **Chips region** — `div className="flex flex-wrap gap-2"` containing one `Badge variant="secondary"` per existing keyword, each with a trailing `X` remove button. When the list is empty, render the `Empty` primitive (heading + body copy above) INSTEAD of the chip row.
3. **Add row** — a `<form onSubmit>` with a `Field`: `FieldLabel` ("Nova palavra-chave") + `Input` (`maxLength={60}`, `placeholder="Ex.: uber"`, `aria-invalid` bound to the inline error) + a primary `Button type="submit"` ("Adicionar"). The input + button sit on one row (`flex gap-2`), with `FieldError` below.
4. `DialogFooter` → a single `DialogClose` rendering `Button variant="outline"` **"Fechar"** (NO "Salvar" — there is no batch save; every action already persisted).

**Persistence (LOCKED — immediate per action, NOT batch-on-close):**
- Add: submit (Enter or click "Adicionar") → `startTransition(async () => addKeyword(categoryId, value))` → on `{ok:true}` toast success + clear+refocus the input; on duplicate toast.info; on `{error}` toast.error. The server action calls `revalidatePath('/categorias')` so the menu count `(N)` updates.
- Remove: click the chip `X` → `startTransition(async () => removeKeyword(keywordId))` → on success toast + the chip disappears (via revalidate / local optimistic removal); on `{error}` toast.error and the chip stays.
- The dialog reads its keyword list from props seeded by the RSC fetch; after `revalidatePath` the server re-renders the page and the refreshed list flows back in. (Whether the dialog also keeps a local optimistic copy for instant feedback is Claude's discretion, as long as the persisted source of truth is the server.)

**Keyboard & focus (LOCKED):**
- **Enter-to-add:** the add row is a real `<form>`; pressing Enter in the input submits → adds the keyword (do NOT also close the dialog). After a successful add, **clear the input and return focus to it** so the user can type the next keyword without reaching for the mouse.
- **Dialog focus management:** on open, focus moves into the dialog and lands on the keyword Input (the primary task is typing a new term) — set via `autoFocus` on the Input or the Dialog's `initialFocus`. Focus is trapped within the dialog (Base UI default). On close, focus returns to the triggering menu item / row actions button (Base UI default — do not override).
- **Esc** closes the dialog (Base UI default). Closing does not require/trigger a save (already persisted).
- Each chip `X` is a real `<button type="button">`, keyboard-focusable and Enter/Space-activatable, with the `aria-label` above. Tab order: chips (each with its X) → input → "Adicionar" → "Fechar".

**Loading state.** While a server action is in flight (`isPending`), disable the "Adicionar" button and show its pending label ("Adicionando…"), mirroring `categoria-form.tsx`'s `{isPending ? 'Salvando…' : 'Salvar'}`. Do NOT block the whole dialog or show a spinner overlay — actions are fast and the disabled button is sufficient feedback. Removing a chip may optimistically grey/disable that chip's `X` during its transition.

**No reflow / no Phase-20 leakage.** This dialog shows ONLY the CRUD surface. No matching preview, no "this keyword matched N transactions", no `source="palavra-chave"` affordance, no auto-suggested keywords — all explicitly deferred to Phase 20 / KW-F1.

## Accessibility Basics

- **Color never the sole signal:** chips carry text; the remove control carries `aria-label="Remover palavra-chave {keyword}"`; the count is a literal number. Nothing relies on hue alone.
- **Chip-remove buttons** are real `<button type="button">` (not clickable spans), focusable, Enter/Space-activatable, each with a unique descriptive `aria-label` including the keyword.
- **Add input** has an associated `FieldLabel` (`htmlFor`/`id`), `aria-invalid` bound to the inline error, and an Enter-submits form so it is fully keyboard-operable.
- **Focus:** visible gold ring (`--ring`) on the input, "Adicionar", "Fechar", and each chip `X` (inherited global focus treatment). Focus moves to the input on open and returns to the trigger on close; focus is trapped while open.
- **Live feedback:** success/duplicate/error use `sonner` toasts (already themed + `aria-live` by the toaster); the inline `FieldError` is associated to the input for screen-reader announcement on validation failure.
- **Touch targets:** chip `X` hit area meets the inherited mobile touch guidance (wrap the `size-3` glyph so the tappable region is comfortable on touch).
- Honors `prefers-reduced-motion` (inherited — chip add/remove is opacity/layout only, no decorative motion).
