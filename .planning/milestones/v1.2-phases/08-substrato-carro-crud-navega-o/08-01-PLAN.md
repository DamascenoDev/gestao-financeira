<objective>
Front-load the entire irreversible data substrate of the Carro module: the `carros` and `abastecimentos` tables, the nullable `transactions.carro_id` column, the two `security_invoker` consumption views, plus uniform RLS / grants / indexes — then apply the migration to the local stack and regenerate the typed client with no drift, with Wave-0 tests proving cross-user isolation and view non-leak.

Purpose: This is the BLOCKING substrate every later Carro slice (CRUD UI here in Plan 03, etiquetagem in Phase 9, abastecimento+consumo in Phase 10, detail+chart in Phase 11) depends on. Schema is the one thing that cannot be retrofitted cheaply — the two project invariants (integer-cents money, RLS `auth.uid()=user_id`) and the abastecimento cost XOR rule (D2) are pinned in SQL from the first byte. Verification gates (build/tsc) are off-config and yield false positives unless the migration is applied and types regenerated FIRST.

Output: `supabase/migrations/0027_carros.sql`, a regenerated `src/types/database.types.ts`, and two Wave-0 integration tests (`tests/carro-rls.test.ts`, `tests/carro-view-leak.test.ts`).
</objective>

    Table `public.carros`: `id uuid primary key default gen_random_uuid()`, `user_id uuid not null references auth.users(id) on delete cascade`, `apelido text not null`, `modelo text`, `placa text`, `ano int`, `combustivel_padrao text`, `is_archived boolean not null default false`, `created_at timestamptz not null default now()`. Index `carros_user_idx on public.carros (user_id)`.

    Table `public.abastecimentos`: `id uuid primary key default gen_random_uuid()`, `user_id uuid not null references auth.users(id) on delete cascade`, `carro_id uuid not null references public.carros(id) on delete cascade`, `occurred_on date not null`, `odometro_km int not null check (odometro_km > 0)`, `litros numeric(7,3) not null check (litros > 0)` (litros is NOT money — numeric, never bigint cents), `tanque_cheio boolean not null`, `combustivel text`, `transaction_id uuid references public.transactions(id) on delete set null`, `amount_cents bigint check (amount_cents is null or amount_cents > 0)`, `note text`, `created_at timestamptz not null default now()`. Add the cost XOR CHECK (D2) exactly: `check ((transaction_id is not null and amount_cents is null) or (transaction_id is null and amount_cents is not null))` — exactly one cost source, never both, never neither. Add a partial unique index `abastecimentos_transaction_uniq on public.abastecimentos (transaction_id) where transaction_id is not null` (one lançamento links at most one abastecimento). Index `abastecimentos_user_idx on public.abastecimentos (user_id)` and `abastecimentos_carro_idx on public.abastecimentos (carro_id)`. (No UI this phase — substrate only.)

    ALTER `public.transactions`: `add column if not exists carro_id uuid references public.carros(id) on delete set null` (nullable; D4 — purely additive tag, must NOT alter category/metas accounting; SET NULL so deleting a carro unlinks rather than deletes lançamentos). Index `transactions_carro_idx on public.transactions (carro_id)`.

    RLS for `carros` and `abastecimentos` (uniform 0025 shape): `enable row level security`; `grant select, insert, update, delete ... to authenticated, service_role`; `drop policy if exists` then `create policy "own carros"`/`"own abastecimentos" for all to authenticated using ((select auth.uid()) = user_id) with check ((select auth.uid()) = user_id)`. Do NOT add a new RLS policy for transactions (the existing 0005 policy already scopes the whole row including the new column).

    View `public.v_abastecimento_consumo with (security_invoker = true)`: per carro, ordered by `odometro_km`, using window `lag()` over the tanque_cheio intervals — `km_rodados` = odômetro atual − odômetro do último abastecimento tanque-cheio; `litros_intervalo` = Σ litros desde o último tanque cheio; `km_por_litro` = km_rodados ÷ litros_intervalo (guard /0 → null, the 0014/0026 sem-receita pattern); `custo_intervalo_cents` = Σ cost using `coalesce(t.amount_cents, a.amount_cents)` (transaction cost when linked, else manual) joining transactions; `reais_por_km` = custo_intervalo ÷ km_rodados (guard /0). Expose `user_id` so RLS-via-invoker scopes it. `grant select ... to authenticated`.

    View `public.v_carro_resumo with (security_invoker = true)`: per carro — `km_por_litro_medio` (avg of the tanque-cheio interval km/l from v_abastecimento_consumo), `reais_por_km_medio`, `preco_litro_medio_cents`, `gasto_total_cents` = `Σ transactions.amount_cents where carro_id = carros.id`, `gasto_mes_corrente_cents`. Expose `user_id`. `grant select ... to authenticated`. Build the consumption averages off v_abastecimento_consumo so the interval logic lives in one place.

    Top-of-file comment must state: integer-cents money, litros is numeric (not money), security_invoker mandatory (a DEFINER view leaks every user's rows), carro_id is a non-accounting additive tag (D4), XOR CHECK is the single-source cost rule (D2). No fenced literals beyond the SQL file itself.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated user → Postgres (RLS) | Each user's queries must see only their own carros/abastecimentos/carro_id rows |
| view consumer → view definer | A view runs as DEFINER unless `security_invoker = true`, bypassing the caller's RLS |
| client-supplied FK → FK target row | A forged carro_id / transaction_id satisfies a (non-RLS-aware) FK globally |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Information Disclosure | carros / abastecimentos tables | mitigate | Uniform RLS `(select auth.uid()) = user_id` USING + WITH CHECK + grants only to authenticated/service_role; proven by tests/carro-rls.test.ts (User B reads zero) |
| T-08-02 | Information Disclosure | v_abastecimento_consumo, v_carro_resumo | mitigate | `with (security_invoker = true)` on both views so they inherit the caller's RLS; a DEFINER view would leak every user's rows; proven by tests/carro-view-leak.test.ts |
| T-08-03 | Tampering | abastecimentos cost source | mitigate | DB-level XOR CHECK (transaction_id XOR amount_cents) + partial unique index — DB rejects both/neither/double-link regardless of app code; proven by carro-rls negative-constraint block |
| T-08-04 | Tampering | transactions.carro_id accounting impact | accept | carro_id is nullable + non-accounting (D4) and no view/policy keys budget/metas off it; tagging cannot alter category totals by construction (later-phase server writes re-derive ownership) |
| T-08-05 | Information Disclosure | service-role key in client bundle | accept | This plan ships only SQL + generated types + tests; no client code imports the service-role key (the secret-bundle audit, SEC-01, remains green; admin client is unused here) |
</threat_model>
