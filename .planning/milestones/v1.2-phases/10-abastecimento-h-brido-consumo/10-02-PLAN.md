<objective>
Build the typed, validated server boundary for abastecimentos (CAR-03) and the consumption presentation helpers (CAR-04). This is the contract the UI (10-03) calls: a Zod schema that enforces the cost-source XOR, server actions that re-derive ownership of BOTH carro_id AND the linked transaction (dual IDOR), set carro_id on the linked transaction so fuel shows in the carro's spend, and a pure helper that derives preco_litro / km/l / R$/km for display.

Purpose: The cost-source XOR and the dual ownership re-derive are the two security-critical invariants of this phase. They live at the server boundary, fully tested, BEFORE any form exists — so the UI is pure wiring over a proven contract. (CAR-03/04; mirrors the actions/transactions.ts carro_id sync + ownership.ts IDOR pattern from Phase 9.)

Output: `src/lib/schemas/abastecimento.ts`, `src/actions/abastecimentos.ts`, `src/lib/carro/consumo.ts`, their unit tests, and the Wave-0 dual-IDOR/XOR/link-sync integration test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → createAbastecimento | Untrusted carroId, transactionId, amountCents, litros, odometroKm cross here; all re-validated + re-owned server-side |
| action → transactions.carro_id FK | Setting carro_id on a linked transaction must target ONLY the caller's own, unlinked transaction |
| action → abastecimentos.transaction_id FK | A linked transaction must be the caller's own and not already linked (1:1) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04 | Tampering | cost source (transaction_id vs amount_cents) | mitigate | abastecimentoSchema XOR refine (both/neither reject) at the boundary PLUS the DB cost XOR CHECK from 0027 — defense in depth. Proven by abastecimento.test.ts + abastecimento-action.test.ts. |
| T-10-05 | Elevation of Privilege / IDOR | transactions.carro_id write on link | mitigate | Re-derive transaction ownership (assertOwnedTransaction exactly-1-row) before setting carro_id; a forged/foreign transactionId writes nothing. Proven by abastecimento-action.test.ts (links B's tx → error, B.carro_id stays null). |
| T-10-06 | IDOR | abastecimentos.carro_id FK | mitigate | assertOwnedCarro tri-state re-derive before any insert/update — a forged carroId returns 'Carro inválido.' with no write (the Phase-9 pattern). |
| T-10-07 | Tampering | 1:1 transaction↔abastecimento link | mitigate | Pre-check (select abastecimentos where transaction_id) + the partial unique index from 0027 reject a second link; race maps to the already-linked error. |
| T-10-SC | Tampering | npm/pip/cargo installs | mitigate | No new package-manager installs (uses existing zod, supabase-js, decimal/Intl money helpers). N/A — no [ASSUMED]/[SUS] package introduced. |
</threat_model>
