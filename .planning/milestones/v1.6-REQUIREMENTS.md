# Requirements: Gestão Financeira Pessoal — Milestone v1.6

**Defined:** 2026-06-20
**Milestone:** v1.6 "Classificação fluida & ingestão robusta"
**Core Value:** Subir uma fatura e ver os gastos classificados automaticamente (memória que aprende com cada confirmação) junto com a aderência às metas.

## Milestone v1.6 Requirements

Requisitos deste milestone. Cada um mapeia para exatamente uma fase do roadmap. Numeração de REQ-IDs continua das categorias existentes (KW-06, CLSAI-09, PDF-05, IMP-06 são os últimos usados).

### Classificação fluida — Palavra-chave (KW)

- [x] **KW-07**: Ao confirmar um padrão merchant→categoria na grid de revisão, o usuário recebe a opção **inline** de criar uma palavra-chave para aquele descritor (opt-in, sem criar automaticamente).
- [x] **KW-08**: Em `/categorias`, o usuário vê um painel que analisa padrões já confirmados (`merchant_patterns`) e sugere palavras-chave candidatas, aprovando ou descartando em lote.
- [x] **KW-09**: O usuário pode usar wildcard glob (`*`) em uma palavra-chave (ex.: `UBER*`, `*IFOOD*`) além do match por substring atual; em conflito, a palavra-chave mais específica vence ("maior keyword vence" preservado).
- [x] **KW-10**: A procedência `palavra-chave` é persistida em `transactions.classification_source` (deixa de gravar o coarse `memória`), via widening do CHECK da migration `0020`.

### Classificação fluida — Sugestões em lote (CLSAI)

- [x] **CLSAI-10**: No review grid, o usuário pode aplicar de uma vez todas as sugestões pendentes (memória / palavra-chave / IA) cuja confiança esteja **acima de um limiar**, deixando as de baixa confiança para revisão manual. Sem auto-commit no upload — a ação é explícita do usuário.

### Ingestão robusta — PDF (PDF)

- [x] **PDF-06**: Upload de fatura PDF funciona em **produção** (Vercel): o worker do `pdfjs` está disponível no bundle serverless, sem erro de "worker faltando".
- [x] **PDF-07**: O parser de PDF degrada de forma clara e previsível em entradas ruins (image-only / 0 linhas), sem travar nem produzir linhas silenciosamente erradas — robustez genérica, **sem OCR**.

### Ingestão robusta — Re-import (IMP)

- [x] **IMP-07**: Re-upload do mesmo arquivo é permitido quando a importação anterior **não foi confirmada** (rows que nunca viraram transactions); o `content_hash` só bloqueia o que já foi confirmado.

## Future Requirements

Reconhecidos, mas adiados — não entram no roadmap do v1.6.

### Palavra-chave avançada

- **KW-F (regex)**: Match por regex completo em palavra-chave (além do wildcard glob). Adiado por risco de ReDoS + erros do usuário; reavaliar se o wildcard se mostrar insuficiente.

### PDF avançado

- **PDF-F (per-bank)**: Parser dedicado por banco. Adiado até aparecer um banco real cujo PDF o `getText` extraia mal (sem caso atual; Santander já funciona desde v1.3).

## Out of Scope

Exclusões explícitas, com motivo, para evitar scope creep.

| Feature | Reason |
|---------|--------|
| Regex puro em palavra-chave | Risco de ReDoS + erros de usuário; wildcard glob (`*`) cobre o caso de uso com segurança |
| OCR de PDF image-only | Pesado e com custo de provedor externo; mantido fora do escopo — orientar CSV/OFX para o banco problemático |
| Parser PDF dedicado por banco | Sem banco real falhando hoje; entra só quando houver caso concreto (vira PDF-F) |
| Auto-commit de qualquer classificação | Princípio do produto: confirmação humana no loop — nem keyword, nem IA, nem aplicar-em-lote commitam sozinhos no upload |
| Integração bancária automática (Open Finance) | Mantido out-of-scope do projeto: ingestão é por upload manual |
